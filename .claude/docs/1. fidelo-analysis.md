## Core Entities

### Student / Contact
- **Identity:** id, contact_number, firstname, lastname, email, gender (1=male, 2=female), birthday
- **Location:** nationality (ISO 3166-1 alpha-2), language (ISO 639-1), address, zip, city, country_iso, state
- **Contact:** phone_private, phone_mobile
- **Emergency:** emergency_firstname, emergency_lastname, emergency_phone, emergency_email
- **Billing:** billing_firstname, billing_lastname, billing_phone, billing_email, billing_address, billing_zip, billing_city, billing_country, billingaddress_company
- **Visa/Passport:** passport_number, visa_required, visum_name, visum_tracking_number, visum_passport_number, visum_date_of_issue, visum_due_date, visum_date_from, visum_date_until
- **Other:** referrer_id, student_status_id, social_security_number, matching_allergies, profile_picture, customer_comment, custom_fields

### Booking
The central commercial entity linking student → courses, accommodation, payments, invoices.
- **Core:** id, contact_id, booking_number, document_number, customer_number, inbox
- **Status:** status (e.g. "ready"), confirmed (boolean/timestamp), confirmed_date, checkin, checkout, follow_up
- **Dates:** all_start, all_end, booking_created_original, changed_original
- **Agency (B2B):** agency_id, agency_name, agency_full_name, agency_number, agency_category
- **Sponsorship:** sponsored, sponsor_name, sponsor_id_number, sponsoring_fg_from, sponsoring_fg_until
- **Group:** group, group_number, has_student_app
- **Promotion:** promotion
- **Audit:** creator_id, editor_id, created_original, changed_original

### Course (nested within Booking)
- active, from, until, weeks, name, category, note, flexible_allocation, custom_fields, additional_services
- **Weekly assignments:** week, class, days (array of day numbers), start (time), end (time), teacher
- **Summary fields:** course_units, course_weeks, current_level_intern_short, last_teacher_name, placementtest_level_internal

### Accommodation (nested within Booking)
- active, from, until, weeks, category (Hotel, Host Family, etc.), roomtype (Single, Double, etc.), board (Breakfast, Half-board, etc.), from_time, until_time, note, additional_services
- **Assignments:** from, until, provider, room
- **Summary fields:** accommodation_name, accommodation_room, accommodation_bed, accommodation_street, accommodation_city, accommodation_tel, accommodation_email

### Transfer (nested within Booking)
- date, time, type (1=arrival, 2=departure)
- airline, flight_number, comment

### Holiday (nested within Booking)
- from, until, weeks, type (e.g. "student")
- Tracks how holidays split courses and accommodations (original_id, split_id, original_from, original_until)

### Invoice (nested within Booking)
- filename, number (format: D{YEAR}{SEQ} for paid, P for proforma), type ("brutto"/"netto"), date, created, updated, created_by, updated_by
- **Line items:** description, service_from, service_until, amount, discount, commission, tax, active
- **Key metric:** NADC (Net After Discount & Commission) = amount - discount - commission

### Payment
- **Creation fields:** inquiry_id, school_id, booking_id, payment_date, payment_method_id, payment_amount, payment_comment, payment_firstname, payment_lastname, payment_transaction_code
- **Booking-level summary:** amount (total), amount_initial, open_payment_prior_to_arrival, open_payment_at_school, payments, amount_open, amount_refund
- Can be **allocated** (linked to booking) or **unallocated** (standalone)
- Methods: Bank Transfer, Stripe, TransferMate, TransferMate Escrow, Cash

### Teacher Payment (from GUI2 reports)
- Per-teacher, per-week records: firstname, classname, days, lessons, hours, salary/rate amounts
- Partial-week logic: first week of payroll period counts only Thu/Fri, last week only Mon–Wed
- Integration with Zoho for leave tracking (leave_taken, leave_balance)
- Manual override fields: hours_included_this_month, weekly_pay, leave_hours, sick_days, manager_checked

### Enquiry (pre-booking lead)
- Required: school_id, firstname, lastname, email, corresponding_language
- Optional: nationality, language, birthday, gender, phone_private, comment, referrer_id, student_status_id, custom_fields

## Course Types & Hours

| Category | Session | Hours/Week |
|---|---|---|
| Morning Classes | morning | 15 |
| Morning Classes Plus | morning | 20 |
| Afternoon Classes | afternoon | 15 |
| Afternoon Classes Plus | afternoon | 20 |
| Intensive / GE30 | both | 30 |
| Private Lessons | separate | varies |

**Intensive split rule:** GE30 courses (30 hrs/week) are split 50/50 between morning and afternoon for all financial calculations.

## B2B vs B2C

- **B2C (Direct):** No agent on booking. Revenue = amount - discount.
- **B2B (Agent):** agent/agency fields populated. Revenue = amount - discount - commission. Typical commission ~25%.
- B2B CAC is ~3.3x higher than B2C CAC (773 EUR vs 233 EUR in 2024).
- B2B payback period ~3 months vs < 1 month for B2C.
- Agent gross rate should be ~33% higher than direct rate to offset the ~25% commission.

---

## Financial Metrics

- **NADC** (Net After Discount & Commission) = amount - discount - commission. This is actual school revenue.
- **Hourly rate** = NADC / totalHours (weeks × hoursPerWeek). Calculated per session and channel.
- **2025 hourly rates:** Morning B2C: €6.96/hr | Morning B2B: €4.98/hr | Afternoon B2C: €5.90/hr | Afternoon B2B: €3.43/hr
- **Fee breakdown categories:** course fees, accommodation fees, transfer, insurance, additional course fees, additional accommodation fees, general additional fees, manually entered positions, overpayment
- **Only D-invoices** (paid) count for revenue — proformas (P) and credit notes excluded.

## Fidelo API

### Standard API (Bearer token)
- **Base URL:** `https://ulearn.fidelo.com/api/1.0/ts/` (or `1.1`)
- **Auth:** `Authorization: Bearer <JWT_TOKEN>` — IP whitelist required
- **Limit:** 1000 records per request — must extract month-by-month

| Endpoint | Method | Purpose |
|---|---|---|
| `/bookings` | GET | List bookings (with filters) |
| `/booking/{id}` | GET | Booking detail (courses, accommodation, transfers, holidays, invoices) |
| `/bookings` | POST | Create booking |
| `/bookings/{id}` | PATCH | Update booking |
| `/enquiries` | POST/GET | Create/search enquiries |
| `/enquiries/{id}` | PATCH/GET | Update/get enquiry |
| `/payments` | POST | Create payment (allocated or unallocated) |

**Important:** The `payments` field on bookings is READ-ONLY via PATCH. Payments must be created via `POST /payments` with both `inquiry_id` and `booking_id`.

### GUI2 API (Session-based, internal reports)
- **Auth:** Session cookies via programmatic login to `/api/1.0/login`
- **Endpoint hashes:**
  - `e012597a49e0b3d0306f48e499505673` — Payment Detail
  - `4e289ca973cc2b424d58ec10197bd160` — Payments summary
  - `2962d7de1e4d84081eac5d2e261bb197` — Pay Teachers
- **Response:** `{ data: { head: [...], body: [...] } }` where body items have `{ id, style, items: [...] }`

## Data Sync Architecture (existing scripts)

1. **Booking Sync** → Fidelo API → MySQL `bookings` + `bookings_ext` tables (split for row-size limits)
2. **Payment Sync** → GUI2 endpoint → MySQL `payments` table
3. **Payment Detail Sync** → GUI2 endpoint → MySQL `payment_detail` table (composite key: MD5 of invoice+date+amount+student+payment_id)
4. **Teacher Payment Sync** → GUI2 session + instance hash → MySQL `teacher_payments` table (preserves manual edits on re-sync)
5. **CSV Import** → Fidelo CSV exports → `payment_detail` table

All sync scripts handle: HTML tag stripping, UTF-8 encoding, DD/MM/YYYY → YYYY-MM-DD conversion, currency symbol cleanup, dynamic column creation via ALTER TABLE.

### Key Pain Points Driving Migration
1. **Teacher payroll** — can't use Pay Provider for teachers, manual workaround required
2. **Reporting** — so bad that government reports are often guessed
3. **Class scheduling UI** — hard to read, overly complicated
4. **Permissions** — someone gave everyone all permissions, entire access control is broken
5. **Agency commission** — can't assign different rates easily, forced into categories
6. **Human middleware** — staff manually copying data between HubSpot and Fidelo instead of syncing
7. **Invoicing** — should draw from HubSpot deal data rather than re-entering everything
8. **German naming conventions** — confusing terminology throughout

### Key Architecture Insight: HubSpot as Pre-Sale Source of Truth
The transition point is Deal → WON. Before that, HubSpot owns the data. The new SIS should:
- Allow creating a student/enquiry at any pipeline stage (as a lead/inquiry)
- Convert to full student when deal is won
- Pull pricing/invoice data from the existing HubSpot deal rather than re-entering
- Sync agency info from HubSpot rather than maintaining a separate CRM

## The 5 Systems Inside Fidelo

Fidelo is really 5 systems pretending to be 1:

1. **Student CRM** — student record, contacts, visa/status, documents, communication log, linked bookings. Straightforward.
2. **Bookings Engine** — the commercial heart. One student → many bookings, with dates, gaps/pauses/holidays, course type, pricing, payment status, accommodation. Manageable if modelled cleanly.
3. **Class Scheduling / Assignment** — the hardest operational piece. Classrooms, time slots, levels, teacher assignment, student placement, weekly/daily granularity, historical changes. Where systems get messy fast.
4. **Accommodation Allocation** — second hardest operational piece. Host family → property → room → bed → occupancy calendar → student placement → provider payment. Must be modelled as inventory with dates, not vague text fields.
5. **Documents / Letters / PDF / Email** — visa letters, booking confirmations, invoices, QR verification. Not the hard part — only feels hard because it's attached to everything else.

## Build Order

### Phase 1 — Foundation (Schema)
Core entities and relationships only. Every operational thing gets its own table. Views are for reporting, not business logic.

### Phase 2 — Bookings First
Student CRUD, booking CRUD, course dates, pricing, payment state, notes/documents, booking timeline. **Everything else depends on this.** If bookings are weak, the whole system is weak.

### Phase 3 — Class Assignment
Class template (level + slot + room + date range) → class session/occurrence (dated) → teacher assignment (start/end datetime) → student class enrollment (booking + class + date range). Not a giant timetable object.

### Phase 4 — Accommodation
Mirrors booking/class logic. Host family → property → room → bed → placement → occupancy calendar → provider payment rule. Each bed = reservable resource over time.

### Phase 5 — Documents / PDF / Email
Booking confirmation, visa letter, accommodation confirmation, email with attachment, QR verification endpoint.

## The Critical Design Decision

Separate these 3 concepts cleanly:

1. **Student** — the person
2. **Booking** — the commercial agreement / enrolment instance
3. **Assignment** — where that booking is operationally placed (class, room, bed, teacher time, accommodation)

Legacy systems muddle these together. Clean separation makes reporting, changes, pauses, reassignments, history, billing, and document generation all dramatically easier.

---

## Implications for New SIS Schema

1. **Invoice line items** need granularity — amount, discount, commission tracked per line, not just booking level
2. **B2B/B2C channel** is fundamental to financial reporting — must be a first-class concept
3. **NADC** should be a computed/stored field for reporting
4. **Course type → hours mapping** is core business logic for rate calculations
5. **Teacher payroll** needs partial-week logic and leave integration
6. **Accommodation** is a time-based resource (provider → property → room → bed → date-range placement)
7. **Audit trail** on everything — Fidelo tracks creator_id, editor_id, created, changed on most entities
8. **Payment allocation** — payments can exist unlinked, then be allocated to bookings
